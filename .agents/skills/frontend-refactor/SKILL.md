# Frontend Refactor Skill

> ⚠️ **核心原则**：步骤必须顺序执行，**禁止跳步**，**禁止依赖猜测**，**禁止复用原有代码片段**。
>
> 重构产物默认在**原文件同级目录**下新增文件（或新增目录），不覆盖原文件。

---

## 工作流总览

```
第 1 步：识别重构目标
    ↓
第 2 步：分析现有代码逻辑
    ↓
第 3 步：搜索并加载可用规范 Skill
    ↓
第 4 步：整合分析结果，生成重构清单
    ↓
第 5 步：等待用户确认重构清单
    ↓
第 6 步：拆分重构批次
    ↓
第 7 步：逐批执行重构（全新生成，不复用原代码）
    ↓
第 8 步：自检与输出
```

---

## 第 1 步：识别重构目标

**做什么**：明确本次重构的范围与类型。

| 场景 | 判定 |
|------|------|
| 用户指定了具体文件路径 | 单文件重构 |
| 用户指定了目录路径 | 目录级重构（整个目录） |
| 用户描述了某个功能模块 | 根据上下文定位文件/目录 |
| 用户未明确指定目标 | 询问用户，**不得猜测** |

**输出**：
- `REFACTOR_TARGET`：目标文件路径列表 或 目录路径
- `REFACTOR_TYPE`：`single-file` / `multi-file` / `directory`
- `OUTPUT_STRATEGY`：
  - 单文件 → 在同级目录新增 `<原文件名>.refactored.<ext>` 或用户指定名称
  - 目录 → 在同级新增 `<原目录名>-refactored/` 目录

> 若目标不明确，**停止并询问用户**，不得继续。

---

## 第 2 步：分析现有代码逻辑

**做什么**：深度阅读目标代码，梳理其完整逻辑，**不做任何修改**。

### 2.1 读取目标文件

使用 `read_file` 工具读取所有目标文件的完整内容。若为目录，先用 `list_dir` 列出所有文件，再逐一读取。

### 2.2 梳理逻辑清单

对每个文件，提取以下维度：

| 维度 | 说明 |
|------|------|
| **组件/模块职责** | 该文件/组件的核心功能是什么 |
| **数据流** | Props / Emits / Store / API 调用链路 |
| **状态管理** | 本地 ref/reactive、Pinia store、computed 依赖 |
| **副作用** | watch、onMounted、定时器、事件监听等 |
| **UI 结构** | 模板层级、条件渲染、列表渲染、插槽 |
| **样式规则** | scoped/全局、CSS 变量、响应式断点 |
| **依赖引用** | 外部组件、工具函数、API 接口、类型定义 |
| **已知问题** | 代码异味、重复逻辑、过时写法、潜在 Bug |

### 2.3 记录分析结论

将分析结论以结构化形式记录到上下文，供后续步骤使用。**不得跳过此步骤直接进入重构**。

---

## 第 3 步：搜索并加载可用规范 Skill

**做什么**：在重构前，主动搜索当前项目中可用的规范类 Skill，确保重构产物符合项目标准。

### 3.1 搜索策略（按优先级）

**优先级 1：项目级 Skill（最高优先级）**

查阅当前上下文中 `<available_skills>` 列表，筛选出与本次重构目标技术栈相关的项目级规范 Skill（如前端框架规范、组件库规范、编码规范等），逐一读取其 `SKILL.md`。

**优先级 2：全局 Skill 且分析项目同类代码**

若项目级 Skill 不存在或不适用，同时执行以下两项：
1. 从 `<available_skills>` 中筛选全局通用规范 Skill（如 `vue-best-practices`、`mdesign-spec-skill`、`vue` 等），读取并提取相关规范
2. 使用 `codebase_search` 在项目中搜索**同类型的非重构目标代码**（同类组件、同类页面、同类工具函数），提取项目中已有的编码模式、命名规范、组件结构作为参考基准，并明确标注：「本次重构参考基准来源于项目同类代码」

### 3.2 加载规范内容

对每个适用的 Skill，使用 `read_file` 读取其 `SKILL.md`，提取：
- 组件/API 使用规范
- 命名约定
- 禁止使用的写法
- 推荐的代码模式

### 3.3 整合规范摘要

将所有规范要点整合为「重构规范摘要」，格式如下：

```
【规范来源】[实际使用的 Skill 名称，如无则填写"项目同类代码"]
【组件规范】[适用的组件库及使用规范]
【命名规范】...
【禁止写法】...
【推荐模式】...
```

> 若无任何规范可用，**必须明确说明**，并列出从项目代码中提取的参考模式。

---

## 第 4 步：整合分析结果，生成重构清单

**做什么**：将第 2 步的代码分析 + 第 3 步的规范摘要整合，生成结构化的重构清单。

### 重构清单格式

```markdown
## 重构清单

### 基本信息
- **重构目标**：[文件/目录路径]
- **重构类型**：[single-file / multi-file / directory]
- **输出位置**：[新文件/目录路径]
- **规范来源**：[使用的 Skill 名称 或 "项目同类代码"]

---

### 逻辑保留清单（必须在重构中体现）
| # | 逻辑描述 | 原代码位置 | 重构方案 |
|---|---------|-----------|---------|
| 1 | ... | L12-L34 | ... |
| 2 | ... | ... | ... |

---

### 重构改进项
| # | 问题描述 | 改进方案 | 规范依据 |
|---|---------|---------|---------|
| 1 | 使用了过时的 Options API | 改为 Composition API + `<script setup>` | vue-best-practices |
| 2 | ... | ... | ... |

---

### 新增/调整项
| # | 内容 | 原因 |
|---|------|------|
| 1 | 新增 TypeScript 类型定义 | 项目规范要求 |
| 2 | ... | ... |

---

### 不变项（保持原有逻辑，仅重新实现）
- 业务逻辑 A：...
- 数据结构 B：...

---

### 风险提示
- [ ] [潜在风险描述，如：依赖了全局变量 X，需确认重构后仍可访问]
```

---

## 第 5 步：等待用户确认重构清单

**做什么**：将重构清单呈现给用户，**等待明确确认后才能继续**。

呈现清单后，说明：

> 「以上是本次重构的完整清单。请确认：
> 1. 逻辑保留清单是否完整？
> 2. 改进项是否符合预期？
> 3. 输出位置是否正确？
>
> 确认后我将开始重构，重构产物为全新文件，不会修改原文件。」

**禁止在用户确认前开始生成任何重构代码。**

若用户要求修改清单，更新清单后重新等待确认。

---

## 第 6 步：拆分重构批次

**做什么**：将确认后的重构清单拆分为若干批次，每批次代码量可控，避免单次生成过多内容导致质量下降或上下文溢出。

### 6.1 拆分原则

| 维度 | 规则 |
|------|------|
| **单文件** | 按功能区块拆分：① 类型/Props/Emits → ② 状态与逻辑 → ③ 模板 → ④ 样式 |
| **多文件** | 每个文件为一个独立批次，按 6.3 执行顺序排列 |
| **目录** | 先拆分文件列表，再按文件逐批执行 |
| **代码量上限** | 每批次生成代码不超过 **150 行**；超出则继续拆分 |

### 6.2 批次计划格式

拆分完成后，向用户展示批次计划，格式如下：

```
【批次计划】共 N 批

批次 1：[文件名 / 区块描述]（预估 ~XX 行）
批次 2：[文件名 / 区块描述]（预估 ~XX 行）
...
批次 N：[文件名 / 区块描述]（预估 ~XX 行）

准备好后请回复「开始」，我将逐批执行，每批完成后等待你确认再继续。
```

**禁止在用户回复确认前开始生成任何批次代码。**

---

## 第 7 步：逐批执行重构

**做什么**：根据第 6 步的批次计划，**逐批**全新生成重构代码，每批完成后暂停等待用户确认。

### 6.1 核心约束

> ⚠️ **禁止复用原有代码**：原有代码仅作为逻辑参考，所有代码必须重新编写。
>
> 这意味着：
> - 不得复制粘贴原文件的任何代码片段
> - 不得保留原文件的注释（除非注释本身是业务说明）
> - 变量命名、函数结构、模板组织均需按规范重新设计
> - 若原代码有 Bug，重构时应修正

### 6.2 输出位置规则

| 重构类型 | 输出规则 |
|---------|---------|
| 单文件 | 在原文件**同级目录**新增 `<原文件名>.refactored.<ext>` |
| 多文件 | 每个文件按单文件规则处理 |
| 目录 | 在原目录**同级**新增 `<原目录名>-refactored/`，内部结构与原目录保持一致 |
| 用户指定输出路径 | 以用户指定为准 |

**示例：**
```
原文件：src/views/user/UserList.vue
输出：  src/views/user/UserList.refactored.vue

原目录：src/views/user/
输出：  src/views/user-refactored/
```

### 7.3 重构执行顺序

对于多文件/目录重构，按以下顺序执行：
1. 类型定义文件（`.ts` 类型文件）
2. 工具函数 / Composables
3. 子组件（从叶子节点到父节点）
4. 页面级组件
5. 路由/入口文件

### 7.4 每批次完成后的交互

每完成一个批次后，输出：

```
✅ 批次 X/N 完成：[文件名 / 区块描述]

[生成的代码]

---
请确认本批次内容无误后回复「继续」，我将执行批次 X+1。
如需调整，请说明修改意见，我将修正后再继续。
```

**禁止在用户回复「继续」或明确确认前自动执行下一批次。**

### 7.5 每个文件的生成要求

每个重构文件必须包含：
- 完整的 import 语句（按规范排序）
- 完整的组件/函数实现（不得有 `// TODO` 或 `// ...` 占位）
- 必要的类型注解
- 符合规范的注释（关键逻辑说明）

---

## 第 8 步：自检与输出

**做什么**：重构完成后，对产物进行自检，确保质量。

### 7.1 自检清单

对每个重构文件，逐项检查：

**逻辑完整性**
- [ ] 重构清单中的所有「逻辑保留项」均已实现
- [ ] 所有 Props / Emits 接口与原组件保持兼容（或已在清单中说明变更）
- [ ] API 调用逻辑与原代码一致

**规范符合性**
- [ ] 组件使用符合加载的规范 Skill 要求
- [ ] 命名规范符合项目约定
- [ ] 无禁止使用的写法

**代码质量**
- [ ] 无明显语法错误
- [ ] 无未使用的 import
- [ ] 无硬编码的魔法值（应提取为常量）
- [ ] 无重复逻辑（应提取为函数/composable）

**文件完整性**
- [ ] 文件已创建在正确位置
- [ ] 无 `// TODO`、`// ...`、`// 省略` 等占位符

### 7.2 输出摘要

自检完成后，输出以下摘要：

```markdown
## 重构完成

### 产物清单
| 文件 | 位置 | 状态 |
|------|------|------|
| UserList.refactored.vue | src/views/user/ | ✅ 已生成 |

### 主要改动说明
1. **[改动类型]**：[具体说明]
2. ...

### 注意事项
- [需要用户关注的事项，如：需要更新路由引用、需要更新父组件的 import 路径等]

### 验证建议
- [ ] 在开发环境运行并验证功能
- [ ] 对比原文件与重构文件的行为一致性
- [ ] 如确认无误，可替换原文件
```

## 附录：常见重构场景参考

### Vue 2 Options API → Vue 3 Composition API

重构时参考以下映射关系（**仅作逻辑参考，代码需重新编写**）：

| Options API | Composition API |
|-------------|----------------|
| `data()` | `ref()` / `reactive()` |
| `computed` | `computed()` |
| `methods` | 普通函数 |
| `watch` | `watch()` / `watchEffect()` |
| `mounted` | `onMounted()` |
| `props` | `defineProps()` |
| `emits` | `defineEmits()` |
| `$store` | `useStore()` / Pinia composable |

### 组件拆分重构

当单个组件超过 300 行或职责不单一时，建议拆分：
1. 在重构清单中明确拆分方案（父组件 + 子组件列表）
2. 用户确认拆分方案后再执行
3. 输出目录结构与原组件保持一致，新增子组件文件

### 样式重构

- 优先使用 CSS 变量（Design Token）替代硬编码颜色/尺寸
- Scoped 样式优先，避免全局污染
- 使用 BEM 或项目约定的命名规范

---

## 禁止行为清单

> 以下行为在任何情况下均**严格禁止**：

1. ❌ 跳过第 2 步（代码分析）直接进入重构
2. ❌ 跳过第 3 步（规范搜索）直接进入重构
3. ❌ 跳过第 5 步（用户确认清单）直接开始生成代码
4. ❌ 跳过第 6 步（批次拆分）直接一次性生成全部重构代码
5. ❌ 每批次完成后未等待用户确认即自动执行下一批次
6. ❌ 复制粘贴原有代码片段到重构产物中
7. ❌ 在重构产物中留下 `// TODO`、`// ...`、`// 省略` 等占位符
8. ❌ 覆盖原文件（重构产物必须是新文件）
9. ❌ 基于猜测生成代码（所有逻辑必须来源于代码分析或规范文档）
10. ❌ 在未读取规范文档的情况下声称「已按规范重构」
